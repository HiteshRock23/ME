# Supermemory Local Integration

ME uses Supermemory Local as its semantic memory engine. Supermemory handles semantic indexing, vector embeddings, and search retrieval.

## Architecture

The integration follows a decoupled architecture where PostgreSQL is the source of truth, and Supermemory acts as a downstream semantic index.

*   **Django (`CaptureService`)**: Responsible for receiving the memory, saving it to PostgreSQL, and initiating the synchronization.
*   **Supermemory (`SupermemoryService`)**: An abstraction layer that communicates via HTTP with the local Supermemory instance (`http://localhost:6767`). It hides all HTTP requests from the rest of the application.

## Synchronization Flow

When a memory is captured:
1.  **Save to PostgreSQL**: The memory is immediately saved to the database with `sync_status = PENDING`.
2.  **Synchronize**: `CaptureService` calls `SupermemoryService.store_memory()`.
3.  **Success**: If Supermemory returns a document ID, the `Memory` model is updated:
    *   `supermemory_document_id` = `<id>`
    *   `sync_status` = `SYNCED`
    *   `synced_at` = `<timestamp>`
4.  **Failure**: If Supermemory is offline or returns an error, the PostgreSQL transaction is **not** rolled back. The model is updated:
    *   `sync_status` = `FAILED`
    *   `last_sync_error` = `<error details>`

This guarantees that user data is never lost due to external service downtime.

## Data Ownership

*   **Source of Truth:** PostgreSQL (`raw_content`).
*   **Idempotency:** When syncing, we pass a `customId` to Supermemory in the format `me_memory_<id>`. This prevents duplicate documents from being created during retries.

## Retry Strategy & Failure Handling

Failures are handled gracefully. The frontend will not display an error if the synchronization fails, because the raw memory has been successfully saved to PostgreSQL.

To recover failed synchronizations, a Django management command is provided:

```bash
python manage.py sync_memories
```

This command queries all `Memory` records where `sync_status != SYNCED` and attempts to synchronize them with Supermemory. It is safe to run repeatedly, as it relies on the idempotent `customId`.

## Troubleshooting

*   **Supermemory offline**: The `sync_memories` command will print an error during its initial health check.
*   **Memory not in Supermemory**: Check the `sync_status` and `last_sync_error` fields in PostgreSQL.
*   **Duplicate results**: This shouldn't happen due to the `customId` mapping. If it does, ensure that no other service is inserting data into the same Supermemory project.

## Semantic Retrieval Architecture (Milestone 7)

ME supports natural language querying to retrieve memories by meaning.

### Retrieval Flow

1.  **Search Input**: User submits a query in the frontend (`GET /api/search/?q=...`).
2.  **Django API**: `SearchView` delegates the query to `SearchService`.
3.  **Supermemory Lookups**: `SearchService` invokes `SupermemoryService.search()`, mapping to Supermemory's `POST /v3/search` API.
4.  **Ownership Verification**: Supermemory returns a ranked list of `documentId`s (`me_memory_12`). `SearchService` extracts the IDs and verifies ownership by querying PostgreSQL for `Memory` records where `user_id` matches the authenticated user.
5.  **Application Response**: Results are returned to the frontend preserving the semantic ranking, preventing unauthorized data leakage.

### Error Handling
- **Empty Query**: Handled frontend/backend gracefully.
- **Service Offline**: Caught as `SupermemoryError` and returned as a 503 HTTP status.
- **No Results**: Displayed dynamically in the frontend as an "empty state" without reloading.
